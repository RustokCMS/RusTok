# server docs

В этой папке хранится документация модуля `apps/server`.

## Документы

## Mandatory platform baseline

Для `apps/server` обязательный baseline нужно читать в двух слоях.

Platform `Core` modules:
- `rustok-auth`
- `rustok-cache`
- `rustok-channel`
- `rustok-email`
- `rustok-index`
- `rustok-outbox`
- `rustok-tenant`
- `rustok-rbac`

Shared foundation / support crates:
- `rustok-core`
- `rustok-events`
- `rustok-telemetry`
- `rustok-api`

Агентам: считать этот baseline обязательным. Логика tenant-toggle/disable относится только к `Optional` modules; `Core` modules не должны трактоваться как отключаемые.

Дополнительно для baseline:
- `rustok-channel` уже участвует в request pipeline через channel resolution middleware и общий `ChannelContext` в `rustok-api`;
- channel resolution middleware теперь только собирает request facts и применяет domain-owned resolver из `rustok-channel`; current pipeline уже выглядит как `header -> query -> built-in host slice -> tenant-scoped typed policies -> default`, а `default` означает explicit default channel tenant'а;
- server держит только thin transport/wiring для channel management: middleware и REST surface `/api/channels/*`, а domain/service/storage остаются в модуле.

- Cleanup/maintenance: background cleanup task supports `sessions`, `rbac-report`, `rbac-backfill`, `rbac-backfill-rollback` targets (`cargo loco task --name cleanup --args "target=<target>"`); RBAC backfill supports safety controls `dry_run=true`, `limit=<N>`, `continue_on_error=true`, `exclude_user_ids=<uuid,...>`, `exclude_roles=<role,...>` and optional rollback snapshot output `rollback_file=<path>`; rollback target consumes `source=<rollback_file>` and supports `dry_run=true`; rollback now removes only role assignments captured in snapshot (role-targeted revert); app `truncate` hook now performs ordered deletion of server foundation tables (`release`, `build`, `tenant_modules`, `sessions`, `users`, `tenants`).
- Profiles backfill: `cargo loco task --name profiles_backfill --args "tenant_id=<uuid> [dry_run=true] [limit=500] [visibility=authenticated] [emit_events=true]"` создаёт missing profiles для существующих пользователей через `rustok-profiles::ProfileService`, использует customer/user seed-данные для display name/locale и может отдельно публиковать `profile.updated` при rollout downstream consumers.
- RBAC rollout migration helpers удалены из live repo после финального перехода на single-engine `casbin_only`; детали cutover при необходимости фиксируются только в ADR/architecture docs и не являются частью текущего authorization path.
- Rich-text migration job (blog/forum legacy markdown → `rt_json_v1`): `cargo run -p rustok-server --bin migrate_legacy_richtext -- --tenant-id=<uuid> [--dry-run] [--checkpoint-file=...]`; job выполняет конвертацию + server-side sanitize/validation + safe update с retry, выводит метрики `processed/succeeded/failed/skipped`, поддерживает идемпотентный restart через checkpoint и предназначен для tenant-by-tenant rollout (с backup-based rollback на tenant scope).
- Auth/JWT algorithm: `rustok-auth` поддерживает `HS256` (default, симметричный shared secret) и `RS256` (асимметричный, RSA PEM ключи). Алгоритм задаётся через `AuthConfig::algorithm: JwtAlgorithm`; для RS256 используется `AuthConfig::with_rs256(private_pem, public_pem)`. Текущий server bridge (`auth_config_from_ctx`) использует HS256 и читает ключ из `auth.jwt.secret` в loco YAML.
- Cache/Redis URL: Redis URL теперь можно задать в YAML через `settings.rustok.cache.redis_url` — имеет приоритет над env vars `RUSTOK_REDIS_URL` / `REDIS_URL`. `CacheService::from_url(Option<&str>)` — новый фабричный метод.
- Email/transactional: добавлен `TransactionalEmailSender` trait для отправки любых transactional email по template ID (`"{module}/{action}"`). Модули реализуют `EmailTemplateProvider` и регистрируются через `SmtpEmailSender::with_provider()`.
- Channels/WebSocket: добавлен `RustokChannel` trait в `apps/server/src/channels/mod.rs` — единый контракт для WebSocket каналов (path + handle). Не зависит от `loco_rs::controller::channels`; использует собственный auth-handshake.
- Auth/outbound email locale: GraphQL `forgot_password` and REST `POST /api/auth/reset/request` dispatch password-reset emails via `tokio::spawn` (non-blocking), and REST `POST /api/auth/verify/request` dispatches email-verification tokens through the same localized auth mailer path. Delivery errors are logged via `tracing::warn` and do not affect the response. Email transport uses `rustok.email` settings; both built-in auth email families now consume the effective request locale for `loco` and `smtp` providers.
- Auth/password reset: REST `POST /api/auth/reset/confirm` теперь отзывает все активные сессии пользователя (через `revoked_at`), что выравнивает policy с GraphQL reset-путём.
- Auth/user lifecycle: GraphQL `create_user` теперь в одной транзакции создаёт пользователя и назначает RBAC связи (`user_roles`/permissions) через `RbacService::assign_role_permissions`.
- Auth/RBAC resolver: `RbacService` использует модульный `rustok-rbac::RuntimePermissionResolver`; в server остаются adapter-реализации `RelationPermissionStore`/`PermissionCache`/`RoleAssignmentStore` (SeaORM + Moka + service wiring), а публичные write-path операции (`assign_role_permissions`/`replace_user_role`) также идут через модульный resolver API.
- Auth/RBAC resolver: `RbacService::get_user_permissions` использует in-memory cache (TTL 60s) с structured cache hit/miss логами и инвалидацией при изменении relation-ролей пользователя.
- Auth/RBAC consistency: REST RBAC extractors (`extractors/rbac.rs`) используют общую wildcard-семантику `resource:manage` через `rustok_rbac::has_effective_permission_in_set`, без локального дублирования permission-логики.
- OAuth/RBAC consistency: `client_credentials` tokens больше не живут на claim-only роли; server live-resolve'ит `oauth_apps.granted_permissions`, строит из них `SecurityContext` и повторно валидирует доступ в domain services. `scopes` остаются OAuth transport-contract'ом, а не заменой platform RBAC.
- Auth/RBAC observability: `/metrics` публикует `rustok_rbac_permission_cache_hits`, `rustok_rbac_permission_cache_misses`, `rustok_rbac_permission_checks_allowed`, `rustok_rbac_permission_checks_denied`, `rustok_rbac_claim_role_mismatch_total`, `rustok_rbac_engine_decisions_casbin_total`, `rustok_rbac_engine_eval_duration_ms_total`, `rustok_rbac_engine_eval_duration_samples`, а также consistency-gauges `rustok_rbac_users_without_roles_total`, `rustok_rbac_orphan_user_roles_total`, `rustok_rbac_orphan_role_permissions_total`, `rustok_rbac_consistency_query_failures_total`, `rustok_rbac_consistency_query_latency_ms_total`, `rustok_rbac_consistency_query_latency_samples`.
- Dev onboarding: `seed_development` creates/updates an idempotent demo tenant (`demo`), demo users, and enables core modules for local environments.
- Admin UI embedding: embedded `/admin` assets are compiled only with Cargo feature `embed-admin-assets` (disabled by default for CI/check environments without frontend artifacts). When enabled, build `apps/admin/dist` before compiling `apps/server`; when disabled, `/admin/*` returns `503 Service Unavailable` with an explicit message that embedding is disabled.
- Ecommerce transport: live REST surface для commerce теперь ограничен `/store/*` и `/admin/*`; compatibility-слой `/api/commerce/*` удалён из runtime router и OpenAPI. Текущий admin commerce surface уже включает `products`, paginated `orders`, `orders/{id}`, explicit order lifecycle endpoints, list/detail/lifecycle endpoints для `payment-collections` и `fulfillments`, а также `shipping-options`/`shipping-profiles` list/show/create/update/deactivate/reactivate; GraphQL schema держит parity-read/lifecycle surface над теми же commerce services, включая admin list queries для payment/fulfillment/shipping-options/shipping-profiles и lifecycle mutations, а storefront GraphQL уже выровнен по discovery (`regions`/`shipping-options`), cart lifecycle, tri-state context patch, customer-owned `me/order` read path и checkout/payment mutations.
- Ecommerce channel-awareness: storefront commerce surface теперь использует request `ChannelContext` как runtime input. `/store/*` и storefront GraphQL уважают `channel_module_bindings`; catalog/shipping visibility и storefront inventory availability могут ограничиваться metadata-based allowlist по `channel_slug`, а cart/checkout path опирается на cart channel snapshot и не пропускает hidden products, hidden shipping options или остатки, скрытые для канала. Channel-aware price resolution пока не считается закрытым и перенесён в Pricing 2.0, чтобы не смешивать storefront availability с pricing semantics.
- Ecommerce deliverability domain: shipping-profile compatibility больше не живёт только как metadata helper. `commerce` уже держит typed registry `shipping_profiles`, typed `products.shipping_profile_slug` / `product_variants.shipping_profile_slug`, snapshot `shipping_profile_slug` в cart/order line items, cart `delivery_groups[]`, typed `shipping_selections[]` и checkout `fulfillments[]`; legacy singular `selected_shipping_option_id`, `shipping_option_id` и `fulfillment` остаются только single-group compatibility shim'ами.
- Ecommerce checkout preflight: перед side effects checkout теперь валидирует stale shipping-profile snapshots, per-group shipping selections и delivery compatibility, отпускает `checking_out` lock на reject path и не создаёт payment/order artifacts до успешного прохождения preflight.
- Ecommerce verification: для live-контракта используются обычные contract/smoke tests по `/store/*`, `/admin/*`, GraphQL parity и schema smoke, без отдельного rollout/deprecation слоя.
- Build pipeline: `BuildService::request_build` now publishes `BuildRequested` via configurable `BuildEventPublisher`; `EventBusBuildEventPublisher` maps it to `DomainEvent::BuildRequested`, while default noop publisher logs skipped dispatch when no runtime wiring is provided.
- Build pipeline: `BuildExecutor` now runs the full manifest-derived plan, not only the server cargo build: `apps/admin` is built with `trunk build`, `apps/storefront` as `cargo build -p rustok-storefront`, and the serialized execution plan is reused by release publishing.
- Marketplace registry V1: `apps/server` now exposes a reference read-only catalog contract at `GET /v1/catalog` plus slug-addressable detail `GET /v1/catalog/{slug}`, both with backward-compatible legacy aliases under `GET /catalog`; the list endpoint now also supports optional `search`, `category`, and `tag` query filters plus `limit` / `offset` paging with `X-Total-Count` for read-only discovery without changing the response body shape, and the resulting list is now deterministically sorted by `slug` so external paging stays stable. Payload is schema-versioned, normalized for `source="registry"`, carries module `name`/`description` metadata for registry-only UX, now also preserves manifest-driven `category`/`tags` plus optional `icon`/`banner`/`screenshots` metadata, trims empty `publisher`, and drops malformed `checksum_sha256` values before those publish signals reach GraphQL/Admin UI. `ManifestManager` now enforces a basic metadata policy for catalog entries (`description >= 20`, `icon` must be absolute `http(s)` `.svg`, `banner`/`screenshots` must be absolute `http(s)` image URLs). Both endpoints now emit `ETag` plus `Cache-Control: public, max-age=60` and honor `If-None-Match` with `304 Not Modified`, so external consumers can treat the reference registry as a cache-friendly read-only host API; those response headers are now explicitly documented in generated OpenAPI as part of the host API surface. A dedicated read-only host can now be booted from the same binary via `settings.rustok.runtime.host_mode = "registry_only"` or env override `RUSTOK_RUNTIME_HOST_MODE=registry_only`: in this mode `apps/server` keeps `health` / `metrics` / `swagger` / registry routes, forces `HeadlessApi` deployment surfaces, skips background workers and optional module route registration, does not expose GraphQL/auth/MCP/admin/storefront shells, aligns `/health/ready` with that reduced surface instead of requiring event/search/module runtime checks that are intentionally absent, publishes a reduced OpenAPI document that advertises only the actually exposed read-only host routes, and correctly wires `ModuleRegistry` into the reduced router so `/health/ready` and `/health/modules` remain live instead of failing on missing runtime extensions. Local deployment smoke for this host mode is now part of `scripts/verify/verify-deployment-profiles.sh`, and the operator runbook in [`health.md`](./health.md) now also fixes the canonical production shape for `modules.rustok.dev`: `headless-api` build profile + `registry_only` runtime host mode + V1 list/detail/cache acceptance + negative checks for V2 write routes.
- Deployment/runtime contract: `DeploymentProfile` and `settings.rustok.runtime.host_mode` are independent axes. `DeploymentProfile` describes build/deploy surfaces (`monolith`, `server-with-admin`, `server-with-storefront`, `headless-api`), while `host_mode` describes the runtime-exposed API surface (`full`, `registry_only`). `registry_only` is intentionally a read-only V1 runtime mode, not a replacement for `HeadlessApi`; future V2 publish/governance routes must remain API-first and available in `full` hosts regardless of whether Leptos admin/storefront are embedded.
- Marketplace registry V2 mutation scaffold: full hosts now expose `POST /v2/catalog/publish`, `GET /v2/catalog/publish/{request_id}`, `PUT /v2/catalog/publish/{request_id}/artifact`, `POST /v2/catalog/publish/{request_id}/validate`, `POST /v2/catalog/publish/{request_id}/approve`, `POST /v2/catalog/publish/{request_id}/reject`, `POST /v2/catalog/owner-transfer` and `POST /v2/catalog/yank` as the current live publish/governance cut. `POST /v2/catalog/publish` creates a request with sync metadata validation and returns `request_id`/`status`; artifact upload now only persists the bundle and moves the request into `submitted`, while validation is triggered as a separate lifecycle step through `/validate` instead of being hard-wired into the upload path. The `/validate` endpoint now acts as a queue boundary: it transitions the request into `validating`, writes an audit event, and returns immediately while the actual bundle verification continues in a background task. That background validator checks `schema_version`, bundle type, module metadata parity, `rustok-module.toml`, crate/UI `Cargo.toml` presence and crate-name/version/license consistency where verifiable, then moves the request into `approved` or `rejected`; only an explicit `approve` action projects the release into V1. `reject` and `owner-transfer` require a non-empty governance reason and surface the resulting moderation state back through status/Admin lifecycle UX. Actor policy is now explicitly split into review actors, owners, and release-management actors: live publish/governance routes use `x-rustok-actor` for audit/governance decisions and `x-rustok-publisher` for persisted owner-binding and release publisher identity. Requested publisher identity is now also stored directly in `registry_publish_requests`, but approve/reject are no longer self-review actions: review paths now require either a registry review actor (`registry:*`, `governance:moderator`, `moderator:*`) or the current persisted slug owner. Persisted slug owners are stored in `registry_module_owners`; once a binding exists, it overrides the old header-pattern fallback for non-governance actors, while governance actors can still create/rebind when needed. Explicit owner transfers now go through `POST /v2/catalog/owner-transfer`, allow either the current persisted owner or a registry admin actor to rebind the slug, and write a dedicated `owner_transferred` audit event with previous/new owner identities plus the transfer reason. `POST /v2/catalog/yank` is now stricter than review actions: it allows the registry admin, the current persisted owner, or the published release actor, rather than every generic moderator. Persisted audit events are now stored in `registry_governance_events` for request creation, artifact upload, validation queue/pass/fail, explicit approve/reject, owner binding, owner transfer, and yank transitions, and those events are exposed back through GraphQL/Admin lifecycle UX as a recent audit trail. Dry-run remains a backward-compatible preview mode, while `registry_only` host mode intentionally stays read-only and does not mount or advertise any V2 write routes.
- Marketplace registry V2 validation orchestration: the current background validator now makes its scope explicit instead of pretending to be a full production gate. Successful `/validate` still marks the request as review-ready only for artifact/manifest contract checks, but now also materializes persisted follow-up stage orchestration for `compile_smoke`, `targeted_tests`, and `security_policy_review`. In practice this means `approved` still means “artifact/manifest contract checks passed and the request is review-ready”, not “all possible release gates were executed inside the registry host”. Re-running `POST /v2/catalog/publish/{request_id}/validate` is still the supported requeue path only for requests whose latest rejection came from automated `validation_failed`; the endpoint still refuses to resurrect manual governance rejects. The queue boundary is no longer only an in-memory `tokio::spawn`: base validation attempts persist to `registry_validation_jobs` with `queued` / `running` / `succeeded` / `failed`, while external follow-up stages now persist separately in `registry_validation_stages` with `queued` / `running` / `passed` / `failed` / `blocked`, attempt numbers, detail, timestamps, and `last_error`. Governance audit events now cover both layers (`validation_job_*` and `validation_stage_*`, with compatibility `follow_up_gate_*` trail retained). There is still no built-in local or remote runner for compile/test/security stages in this increment; operators record those stage results manually through `POST /v2/catalog/publish/{request_id}/stages` or the matching `cargo xtask module stage ...` path. Lifecycle consumers now get both the compatibility `follow_up_gates` summary and a canonical `validation_stages` list in `registryLifecycle`, so Admin can show per-stage state, attempts, and timestamps separately from the base validation result.
- Marketplace registry V1 consumer path: `MarketplaceCatalogService::get_module` and GraphQL `marketplaceModule(slug)` are now detail-aware; they use provider-specific single-module lookup instead of always materializing the full catalog, while preserving provider priority (`local-manifest` stays canonical over `registry` when both expose the same slug). Registry version trails are also normalized before they reach host/UI: `published_at` is canonicalized to RFC3339 UTC when valid, malformed timestamps are dropped, version history is sorted into a stable newest-first trail (`non-yanked` first, then semver/publish date), and GraphQL `latest_version` now falls back to that normalized trail when a registry payload omits top-level `version`.
- Marketplace registry V1 provider chain: `MarketplaceCatalogService::list_modules` now accepts a shared catalog query (`search` / `category` / `tag`) and forwards it through the provider chain; local-manifest providers filter in-process, while `RegistryMarketplaceProvider` now passes those filters to `GET /v1/catalog` and caches filtered responses separately. GraphQL `marketplace(...)` now exposes the same `tag` filter so host/operator discovery can use the same narrowed provider path instead of only the raw HTTP contract.
- Manifest-wired module UI contract: `apps/server` treats `rustok-module.toml` as the source of truth for publishable Leptos surfaces. If a path module contains `admin/Cargo.toml` or `storefront/Cargo.toml` without the matching `[provides.admin_ui].leptos_crate` / `[provides.storefront_ui].leptos_crate`, `ManifestManager` validation and host-side build/codegen now fail explicitly; conversely, declaring a UI provide without the matching sub-crate is also a contract error. The existence of a sub-crate alone is not evidence of host integration. The same manifest layer now also supports optional `[provides.*_ui.i18n]` declarations for package-owned `locales/*.json` / `messages/*.json`; when declared, `ManifestManager` validates locale tags, default locale membership, and existence of every declared bundle file before the module can be treated as a valid live UI surface.
- Module settings schema: `ManifestManager` now supports declarative scalar `options`, shallow nested metadata (`object_keys`, `item_type`), and recursive nested contracts through `properties` / `items` in `[settings]`; schema validation rejects mismatched/default-outside-options cases, unknown object keys, and nested item/property type mismatches. GraphQL exposes shallow metadata plus raw recursive `shape`, and admin UI uses it for schema-driven add-actions in top-level complex object/array editors.
- Server composition root: `apps/server/build.rs` now generates optional module registry wiring, GraphQL schema fragments and HTTP route registration from `modules.toml` into `OUT_DIR`; `apps/server` stays a generic host, while module-specific transport remains inside module crates. Server entry points can now be declared explicitly via `rustok-module.toml` (`[crate]`, `[provides.graphql]`, `[provides.http]`) or, for external/non-path crates, as normalized Rust paths in `modules.toml`; old naming conventions remain only as fallback.
- Release pipeline: `settings.rustok.build.deployment.backend` теперь поддерживает `record_only`, `filesystem`, `http` и `container`; filesystem/container backend публикуют реальные `server_artifact_url`, `admin_artifact_url`, `storefront_artifact_url`, а container backend дополнительно собирает runtime image из release bundle, публикует `container_image` и может вызывать generic `rollout_command` hook без provider-specific логики в `apps/server`.
- Event consumer runtime: long-lived server consumers (`server_event_forwarder`, GraphQL build-progress subscription) follow a shared contract from `rustok-core`: `Lagged -> warn + metric`, `Closed -> explicit stop`, reindex decision path documented in `crates/rustok-outbox/docs/README.md`.
- GraphQL subscriptions: server now exposes `/api/graphql/ws` for subscription transport; browser clients pass `token`, `tenantSlug` and `locale` via `connection_init`, and tenant resolution for this route happens inside the GraphQL websocket handshake instead of the regular header-based tenant middleware.
- GraphQL admin hardening: sensitive admin root fields are now classified from the full document AST inside a shared schema extension before resolver execution. The allow/deny decision no longer depends on client-supplied `operationName`, and persisted query hashes remain telemetry-only metadata rather than an authorization boundary.
- Инвариант transport-слоя: `/api/graphql` и `/api/fn/*` должны поддерживаться параллельно; server functions добавляются как внутренний Leptos path и не заменяют GraphQL API.
- Leptos server functions: server now exposes `/api/fn/*` via `leptos_axum::handle_server_fns_with_context`, прокидывая в Leptos context как минимум `AppContext` и `ModuleRegistry`. `apps/storefront` и `apps/admin` могут использовать этот путь как internal data-layer boundary, но GraphQL `/api/graphql` остаётся таким же обязательным и поддерживаемым transport-слоем; обе реализации должны сосуществовать.
- GraphQL/storefront canonical routing: server now exposes `resolveCanonicalRoute(route, locale)` over the shared GraphQL schema; the resolver uses `rustok-content::CanonicalUrlService` with tenant-scoped canonical/alias tables, follows the shared locale fallback contract (`requested -> en -> first available`), and `apps/storefront` consumes it as SSR preflight before rendering `/modules/{route_segment}`.
- Search GraphQL hardening: public `storefrontSearch` and `storefrontSearchSuggestions` now run behind a dedicated shared rate-limiter namespace, while admin `updateSearchSettings` and `triggerSearchRebuild` emit best-effort audit events through the outbox without turning the mutation path into a hard dependency on the event transport.
- GraphQL schema hygiene: module-owned GraphQL types must be namespace-safe in the shared schema; if a module exposes a domain-specific enum/object, its GraphQL-visible name must not collide with types from other modules.
- GraphQL/DataLoader: shared schema now registers `rustok-profiles::ProfileSummaryLoader` alongside tenant/content loaders, so module-owned resolvers (`profiles`, `blog`, `forum`) can use request-scoped batching/cache for author/member summaries without server-specific SQL joins.
- Tenant cache invalidation: Redis pubsub listener now uses supervised resubscribe with fixed backoff instead of one-shot startup; operational signals go through `rustok_event_consumer_restarted_total` and incident handling stays in `crates/rustok-outbox/docs/README.md`.
- Health/readiness: `tenant_cache_invalidation` is now exposed in `/health/ready`, and current listener state is exported as `rustok_tenant_invalidation_listener_status`; `registry_only` host mode now also propagates explicit `host_mode` / `runtime_dependencies_enabled` signals through `/health/runtime` and runtime guardrail metrics so a dedicated catalog host is self-describing for operators; see [`health.md`](./health.md).
- Auth/session lifecycle: GraphQL `sign_out`, `change_password`, `reset_password` теперь используют soft-revoke через `sessions.revoked_at` (вместо hard delete) и выровнены по поведению с REST (`sign_out` отзывает только текущую сессию, `change_password` — все остальные, `reset_password` — все активные).
- Request trust / tenancy hardening: HTTP request trust is now configured centrally via `settings.rustok.runtime.request_trust`. By default forwarded headers are ignored, tenant `resolution=header` is strict unless `tenant.fallback_mode=default_tenant`, `resolution=subdomain` requires `tenant.base_domains`, and disabled tenants are rejected during middleware resolution with `403`.
- Embedded UI headers: security headers are now surface-aware. Strict CSP remains on `/api/*`, `/metrics`, `/health*`, `/swagger*`, while embedded `/admin` and storefront routes get a UI-compatible CSP. Embedded admin assets also emit `Cache-Control` and `ETag`.
- Rate limiting: the HTTP rate limiter now selects a single policy per request (`oauth` -> `auth` -> `api`) instead of stacking multiple path-prefix middlewares, and IP extraction follows the same trusted-proxy policy as tenant/channel resolution.
- Locale runtime contract: `RequestContext` is the single source of truth for HTTP locale. The effective order is `query -> x-medusa-locale -> cookie -> Accept-Language(q-values) -> tenant.default_locale -> en`, locale-aware responses now emit `Content-Language`, and `tenant_locales` constrain the final selection when configured.
- Locale storage contract: foundation locale columns use the same BCP47-like storage width as runtime (`tenants.default_locale`, `tenant_locales.locale`, `tenant_locales.fallback_locale`), and server-owned `flex` standalone persistence now treats schema copy as localized translation data instead of inline base-row fields.
- Flex attached multilingual storage: generic table `flex_attached_localized_values` is now backed by shared entity/helpers in `crates/flex`. Live donor paths are `user`, `product`, `order`, and `topic`: `user` read-path merges `users.metadata` with locale-specific rows using the effective request locale, `product` resolves locale-aware attached values through the same shared contract in domain read/write flows, `order` does the same for `orders.metadata` while preserving non-Flex operational keys, and `topic` now uses `forum_topics.metadata` as the donor payload with locale-aware Flex values split into parallel attached rows.
- Locale runtime guardrail: package-owned UI bundles and module surfaces may declare translations, but they must not define their own locale negotiation or fallback chain. Server/runtime chooses the effective locale once, host surfaces propagate it, and packages consume that value.
- AuthZ REST contract: permission-gated REST endpoints now return `403` on denied access and `500` on RBAC infrastructure failures instead of collapsing both cases into `401`; denied checks stay in `extractors/rbac.rs`, while `CurrentUser`/RBAC resolver failures are surfaced as `500` from `extractors/auth.rs`.

- Auth/lifecycle extraction: REST handlers и GraphQL mutations для `register/sign_in`, `login/sign_in`, `refresh`, `change_password`, `reset_password`, `update_profile` теперь маршрутизируют бизнес-логику через общий `AuthLifecycleService` (transport adapters остаются тонкими).

- Auth/observability: `/metrics` публикует auth lifecycle counters `auth_password_reset_sessions_revoked_total`, `auth_change_password_sessions_revoked_total`, `auth_flow_inconsistency_total`, `auth_login_inactive_user_attempt_total`; первые два счётчика отражают количество реально отозванных сессий (rows affected), а счётчики ведутся в `AuthLifecycleService` для rollout-периода remediation-плана.

- Auth/error contracts: `AuthLifecycleService` использует типизированные ошибки (`AuthLifecycleError`), а REST/GraphQL делают единообразный transport-specific mapping без дублирования строковых веток.

- Auth rollout controls: канонические release gates, stop-the-line условия и rollback-процедура ведутся централизованно в `docs/architecture/api.md` (раздел «Auth lifecycle consistency и release-gate»); remediation backlog закрыт, в релизах используется operational handoff через `scripts/auth_release_gate.sh --require-all-gates`.
- Auth rollout controls: helper `scripts/auth_release_gate.sh` автоматизирует сбор локального integration evidence (`cargo test -p rustok-server auth_lifecycle` + `cargo test -p rustok-server auth`), всегда формирует markdown gate-report с полями для parity/security evidence и завершает прогон с non-zero exit code при падении любого локального integration auth-среза.
- RBAC/seed consistency: `seed_user` теперь вызывает `RbacService::assign_role_permissions` после создания пользователя, гарантируя наличие `user_roles` для всех seed-пользователей (dev bootstrap).
- MCP management layer: `apps/server` теперь хранит persisted `mcp_clients`, `mcp_tokens`, `mcp_policies`, `mcp_audit_logs` и публикует management API через REST `/api/mcp/*` и GraphQL `mcp*`; это platform capability, а не tenant module.
- MCP runtime bridge: `DbBackedMcpRuntimeBridge` регистрируется в server bootstrap, резолвит plaintext MCP token в `McpAccessContext` на старте stdio-сессии, обновляет `last_used_at` для client/token, пишет runtime allow/deny audit в `mcp_audit_logs` и может работать persisted draft-store backend для Alloy scaffold MCP tools.
- Alloy scaffold draft control plane: `apps/server` теперь также хранит persisted scaffold drafts для Alloy module generation и публикует их через REST `/api/mcp/scaffold-drafts*` и GraphQL `mcpModuleScaffoldDraft*`.
- AI control plane: `apps/server` теперь также хранит persisted `ai_provider_profiles`, `ai_tool_profiles`, `ai_chat_sessions`, `ai_chat_messages`, `ai_chat_runs`, `ai_tool_traces`, `ai_approval_requests` и публикует dual-path contract для `rustok-ai`: GraphQL `ai*` для headless/Next.js и native `#[server]` path для Leptos admin UI; поверх persisted `decision_trace` теперь также доступен in-process diagnostics surface `aiRuntimeMetrics` / `AiManagementService::metrics_snapshot()` для router/run observability, live incremental output идёт через GraphQL subscription `aiSessionEvents(sessionId)` на уже существующем `/api/graphql/ws`, bounded recent stream history доступна через query `aiRecentRunStreamEvents(sessionId?, limit?)`, а recent persisted run history — через `aiRecentRuns(limit?)`.


## Current RBAC contract

- `apps/server` is the only transport/runtime layer that enforces RBAC.
- Runtime modules publish their permission surface through `RusToKModule::permissions()`.
- Runtime module interaction contracts live in `crates/<module>/README.md` under `## Interactions`.
- Server adapters must use `RbacService`, RBAC extractors, or permission-aware `SecurityContext`.
- OAuth `client_credentials` apps are RBAC subjects only through explicit `oauth_apps.granted_permissions`; service tokens do not infer RusToK permissions from OAuth scopes.
- Manual role-based authorization is not part of the live server contract.
